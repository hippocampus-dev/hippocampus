* Next.js on Cloudflare Workers via opennextjs-cloudflare
* Local development (`npm run dev`) supports all Cloudflare bindings (KV, R2, AI) via remote bindings
* Do NOT suggest `npm run preview` for binding access - `npm run dev` works
* Do NOT use `getCloudflareContext` directly in route handlers or Server Components
* Place reusable components in `components/` (not co-located with route segments in `app/`)
* Uses shadcn/ui components in `components/ui/`
* Use `custom-worker.ts` only to re-export Durable Objects / `WorkerEntrypoint` classes that the Next.js handler cannot expose directly
* `custom-worker.ts` (entry importing `.open-next/worker.js`) must be listed in `tsconfig.json` `exclude` — the import target is generated by opennextjs-cloudflare build

## Abstraction Layer

Use Repository pattern for storage, Provider pattern for AI, and Runner pattern for code execution.

| Context | Pattern |
|---------|---------|
| Route handlers / Server Components | `await get*()` factory functions from `lib/` |
| Custom workers (direct env access) | Import `Cloudflare*` classes from `lib/*/cloudflare` |

Factory functions are async and use `getCloudflareContext({ async: true })` internally for SSR/SSG compatibility.

## Storage

| Storage | When to Use |
|---------|-------------|
| KV | Small data (<25MB), fast reads, TTL support |
| R2 | Large content, binary data |

## Dynamic Worker

Passing tail handlers to `WorkerLoader.get({ tails: [...] })`:

| Target | Use |
|--------|-----|
| Tail handler class | `export class X extends WorkerEntrypoint<Env, Props>` implementing `tail(events: TraceItem[])` |
| `tails` array element | `ctx.exports.X({ props: {...} })` — NOT a DO stub |

Retrieve the factory via `(ctx as { exports?: {...} }).exports?.X` from `getCloudflareContext({ async: true })`.
